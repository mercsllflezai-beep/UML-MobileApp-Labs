Лабораторная работа №4
Применение паттернов проектирования: Factory, Decorator, Observer
1. Диаграмма классов (Abstract Factory)
Описание
Диаграмма классов демонстрирует применение порождающего паттерна Abstract Factory для создания различных типов психологических сессий.
Ключевые элементы:
SessionFactory — абстрактный интерфейс, определяющий методы создания сессий и соединений.
StandardSessionFactory — конкретная фабрика для создания обычных сессий (IndividualSession) и защищенных чатов (SecureChatConnection).
EmergencySessionFactory — фабрика для экстренных случаев, создающая сессии кризисной помощи (CrisisSession) и приоритетные линии связи (PriorityHotLineConnection).
Такая структура позволяет системе гибко переключаться между режимами работы (обычный/экстренный) без изменения клиентского кода, обеспечивая расширяемость и независимость от конкретных реализаций.

2. Диаграмма классов (Decorator)
Описание
Диаграмма иллюстрирует структурный паттерн Decorator, используемый для динамического расширения функциональности пользователей.
Ключевые элементы:
UserComponent — базовый интерфейс пользователя.
BasicUser — стандартный зарегистрированный пользователь.
UserDecorator — базовый класс-обертка.
AnonymousModeDecorator — добавляет анонимность (скрывает имя/фото).
PremiumFeaturesDecorator — добавляет доступ к VIP-очереди и расширенным функциям.
DataExportDecorator — добавляет возможность выгрузки истории переписки.
Это позволяет комбинировать поведения во время выполнения (например, «Анонимный Премиум-пользователь») без создания сложных иерархий наследования, соблюдая принцип открытости/закрытости.

3. Диаграмма классов (Observer)
Описание
Диаграмма показывает реализацию поведенческого паттерна Observer для мониторинга безопасности сессий.
Ключевые элементы:
SessionMonitor (Subject) — отслеживает статус сессии и уведомляет подписчиков об изменениях.
Observer — интерфейс для всех наблюдателей.
ModeratorNotifier — получает уведомления о нарушениях и создает тикеты для модераторов.
EmergencyAlertSystem — реагирует на критические статусы (SOS) вызовом экстренных служб.
PsychologistDashboard — обновляет интерфейс специалиста при изменении состояния клиента.
Такой подход обеспечивает слабую связность компонентов: система безопасности реагирует на события автоматически, не требуя прямых вызовов из бизнес-логики чата.

4. Диаграмма последовательности
Описание
Диаграмма последовательности описывает сценарий обработки экстренной ситуации в чате.
Шаги взаимодействия:
Пользователь отправляет сообщение с триггером опасности.
Chat Service передает данные в Session Monitor.
Session Monitor фиксирует статус CRITICAL и вызывает метод notifyObservers().
Emergency System активирует протокол SOS.
Moderator Service создает высокоприоритетный тикет.
Система блокирует автоматические ответы и показывает пользователю контакты помощи.
Диаграмма демонстрирует реактивность системы и четкий порядок действий при возникновении угрозы жизни или здоровью пользователя.

5. Диаграмма состояний
Описание
Диаграмма состояний описывает жизненный цикл Психологической Сессии.
Основные состояния:
Idle — ожидание подключения участников.
Active — активное общение.
Paused — временная приостановка по запросу пользователя.
Critical — режим повышенной опасности (активируется Observer).
Completed — успешное завершение сессии.
Terminated — принудительное завершение администратором или системой безопасности.
Переход в состояние Critical является ключевым точкой входа для механизмов безопасности, обеспечивая безопасность пользователей и контроль качества оказания помощи.

Вывод
В ходе лабораторной работы №4 были спроектированы и визуализированы ключевые механизмы системы HelpHub с использованием паттернов проектирования:
Abstract Factory обеспечила гибкое создание различных типов сессий (обычных и экстренных).
Decorator позволил динамически расширять функциональность пользователей (анонимность, премиум-доступ).
Observer реализовал реактивную систему мониторинга безопасности для быстрого реагирования на критические события.
Диаграммы последовательности и состояний детализировали логику работы системы в сценариях экстренной помощи.
Применение данных паттернов повышает масштабируемость, безопасность и поддерживаемость программного обеспечения.
